# パスの最適化

[![Direct](https://img.shields.io/badge/Direct%20Link-PathCleanupTool.jsx-ffcc00.svg)](https://github.com/swwwitch/illustrator-scripts/blob/master/jsx/path/PathCleanupTool.jsx)

[![Direct](https://img.shields.io/badge/Back%20to%20home-All%20scripts-cccccc.svg)](https://github.com/swwwitch/illustrator-scripts/blob/master/README.md)

---

トレースしたパス、他のアプリから持ち込んだパス、何度も編集を重ねたパス。見た目は問題なくても、アンカーポイントを表示してみると「なぜここに点が？」というものが山ほど乗っていることがあります。直線の途中に等間隔で並ぶアンカー、まったく同じ座標に重なった点、まっすぐな区間なのに引き出されたままのハンドル。

こうしたものは、消しても見た目が変わりません。だからこそ厄介で、**消していいのかどうかを目で確かめられない**わけです。手作業で1点ずつ選んで消すのも現実的ではありません。

そこで、「何がいくつ減るのか」をダイアログ上で先に見せてから実行する形にしました。チェックボックスと許容誤差を動かすたびに、アンカー数・ハンドル数が「実行前 → 実行後」で更新されます。

## 使い方

1. 対象のパスを選択（グループや複合パスのままでOK）
2. スクリプトを実行
3. 情報パネルの予測値を見ながらチェックボックス／許容誤差を調整
4. ［OK］

ダイアログは「削除対象」「変換」の2タブ構成です。**OK を押したときにアクティブだったタブの処理だけが実行されます。** 両方のタブの設定がまとめて適用されることはありません。

## 情報パネル

上部に常時表示され、パス数・アンカーポイント数・ハンドル数を「現在 → 実行後」の形で示します。

この予測値は、DOM を書き換えずに計算しています。対象パスを軽量なモデル（座標の配列）に複製し、その上で実際と同じ処理・同じ実行順をシミュレートして数えているため、実行前にプレビューできても、ドキュメントは一切変更されません。

なお「マド埋め」だけは、パスファインダーの結果を事前に正確に算出できないため、予測値が `-` になります。

## 「削除対象」タブ

| 項目 | 削除されるもの |
| --- | --- |
| 同じ座標のアンカーポイント | **連続して**同じ座標にある点を1つに統合（離れた位置の同座標は対象外） |
| 直線上のアンカーポイント | 前後のアンカーと一直線上にあり、かつハンドルが引き出されていない点 |
| パスと同じ角度のハンドル | 直線とみなせる区間のハンドルをアンカー位置に戻す（見た目は変わらない） |

3項目とも初期状態でオンです。

### 2つの許容誤差

「直線上のアンカーポイント」と「パスと同じ角度のハンドル」には、それぞれ独立した許容誤差（0.01〜3.00）が付いています。数値入力とスライダーが連動し、動かすたびに情報パネルの予測値が更新されます。

値を大きくするほど「直線とみなす」判定が緩くなり、より多く削除されます。同じ「許容誤差」というラベルですが、中身は別物です。

- **アンカー用**：3点の外積で「一直線かどうか」を判定
- **ハンドル用**：両端アンカーを結ぶ直線から、ハンドル先端までの垂直距離（pt）で判定

チェックを外すと、対応する許容誤差の行はディム（グレーアウト）します。

### 実行順が「往復」する理由

3つの処理は、指定した順に1回ずつ流れるわけではありません。実際の実行順はこうなっています。

```
重複アンカー → 冗長アンカー → ハンドル → 冗長アンカー → 重複アンカー
```

ハンドルを直線区間から取り除くと、それまで「ハンドルがあるから消せない」と判定されていたアンカーが、新たに削除対象になります。そこで折り返してもう一度アンカー削除を通す構造です。情報パネルの予測シミュレーションも、まったく同じ順序をなぞっています。

### 端点は消さない

オープンパスの端点は、アンカー削除・重複削除のいずれでも対象外です。ハンドル整理でも、オープンパスの最初のセグメントの始点側ハンドルと、最後のセグメントの終点側ハンドルには触れません。パスの形が壊れないようにするためです。

## 「変換」タブ

3つのグループに分かれていますが、**6つのラジオボタン全体で1つの排他グループ**です（どれか1つだけが選べます）。

### アンカーポイントを変換

| モード | 動作 |
| --- | --- |
| スムーズポイントに | 全アンカーをスムーズポイント化し、ハンドルを付与 |
| コーナーポイントに | 全アンカーをコーナーポイント化し、ハンドルを除去 |

「スムーズポイントに」は、単純に一律のハンドルを付けるのではありません。前後のアンカーまでの距離の 1/3 を基準長として、そこに2つの補正を掛けています。

- **角度補正**：曲がり角が鋭いほどハンドルを短くする（最大35%減）
- **バランス補正**：前後のアンカーまでの距離が偏っているほど短くする（最大25%減）

方向は前後アンカーから求めた接線方向をそのまま使い、8方向などに丸めることはしません。

### アンカーポイントを追加

| モード | 動作 |
| --- | --- |
| 中間に追加 | 各セグメントの中間に1点ずつ追加 |
| 極点を追加 | 曲線の水平・垂直の接線位置（極点）に追加 |

「中間に追加」は Illustrator 標準の「アンカーポイントの追加」メニューを呼んでいます。

「極点を追加」は独自実装です。各セグメントを3次ベジェとして扱い、微分が 0 になる媒介変数 t を解いて、de Casteljau のアルゴリズムで曲線を分割します。**元の曲線の形は変わりません**（分割してもベジェの軌跡は同一）。直線セグメントは対象外です。

極点追加では、パスを作り直すのではなく、同じ PathItem のまま点数を増やして全点を上書きしています。アピアランスを保持するためです。

### その他

| モード | 動作 |
| --- | --- |
| アンカーポイントで分割 | 各セグメントを独立したオープンパスに分割（元のパスは削除） |
| マド埋め | 複合パスを解除して合体し、穴を埋める |

「アンカーポイントで分割」は、元のパスの塗り・線（色・太さ）を各セグメントに引き継ぎます。

「マド埋め」は、グループ化 → 複合パス解除 → ライブパスファインダー（合体）→ アピアランス展開 → グループ解除、というメニューコマンドの連続実行です。**選択内にマド（サブパスが2つ以上ある複合パス）が1つも無い場合は、ラジオボタン自体がディムになり選べません。**

## 対象

PathItem、CompoundPathItem、GroupItem（いずれも再帰的に PathItem を収集）

ロック／非表示のオブジェクトは自動的にスキップされます。判定は親を遡っておこなうため、**オブジェクト自体がロックされていなくても、所属グループやレイヤーがロック／非表示なら対象外**になります。

## 補足

このスクリプトは、ダイアログを開いた時点の選択を配列として固定し、以降はそれを使い回します。情報パネルの予測も、OK 後の実処理も、同じ対象リストを見ているということです。ダイアログ表示中に選択が変わっても、表示と実行がずれません。

処理の直前には、固定しておいたスナップショットから選択を復元しています。「中間に追加」と「マド埋め」はメニューコマンドを使う都合上、選択状態に依存するため、こちらはスキップ対象を除いた PathItem だけを選択し直します。

タブの判定にインデックス（`selection === 0`）を使うと ScriptUI の環境差で誤判定しうるため、タブオブジェクトそのものと比較して分岐しています。

ダイアログの表示位置は、Illustrator を起動している間だけ記憶されます（保存するのは位置のみ。サイズまで復元すると小さすぎて中身が見えなくなることがあるため）。
